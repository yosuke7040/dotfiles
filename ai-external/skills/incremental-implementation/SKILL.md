# 段階的実装

## 概要

薄い垂直スライスで作る。1 つの部品を実装し、テストし、検証してから広げる。機能全体を 1 回で実装することを避ける。各増分は、システムを動作しテスト可能な状態に保つ。この実行規律が、大きな機能を扱える大きさにする。

## 使う場面

- 複数ファイルにまたがる変更を実装する
- タスク分解から新機能を作る
- 既存コードをリファクタリングする
- テスト前におおむね 100 行を超えて書きたくなる

**使わない場面:** スコープがすでに最小である、単一ファイルかつ単一関数の変更。

## 増分サイクル

```
┌──────────────────────────────────────┐
│                                      │
│   Implement ──→ Test ──→ Verify ──┐  │
│       ▲                           │  │
│       └───── Commit ◄─────────────┘  │
│              │                       │
│              ▼                       │
│          Next slice                  │
│                                      │
└──────────────────────────────────────┘
```

各スライスで行うこと:

1. **実装**: 最小の完結した機能片を実装する
2. **テスト**: テストスイートを実行する（存在しなければテストを書く）
3. **検証**: スライスが期待どおり動くことを確認する（テスト成功、ビルド成功、手動確認）
4. **コミット**: 説明的なメッセージで進捗を保存する（atomic commit の指針は `git-workflow-and-versioning` を参照）
5. **次のスライスへ進む**: 続きから進める。やり直さない

## スライス戦略

### 垂直スライス（推奨）

スタックを貫く 1 つの完結した経路を作る:

```
Slice 1: タスクを作成する（DB + API + 基本 UI）
    → テストが通り、ユーザーは UI からタスクを作成できる

Slice 2: タスクを一覧する（query + API + UI）
    → テストが通り、ユーザーは自分のタスクを見られる

Slice 3: タスクを編集する（update + API + UI）
    → テストが通り、ユーザーはタスクを変更できる

Slice 4: タスクを削除する（delete + API + UI + confirmation）
    → テストが通り、CRUD 全体が完了
```

各スライスは、動く end-to-end 機能を届ける。

### 契約ファーストのスライス

バックエンドとフロントエンドを並列開発する必要がある場合:

```
Slice 0: API 契約を定義する（types、interfaces、OpenAPI spec）
Slice 1a: 契約に対してバックエンドを実装 + API テスト
Slice 1b: 契約に合う mock data に対してフロントエンドを実装
Slice 2: 統合して end-to-end でテスト
```

### リスクファーストのスライス

最もリスクが高い、または不確かな部分を先に扱う:

```
Slice 1: WebSocket 接続が動くことを証明する（最大リスク）
Slice 2: 証明済み接続の上にリアルタイムタスク更新を作る
Slice 3: オフライン対応と再接続を追加する
```

Slice 1 が失敗すれば、Slice 2 と 3 へ投資する前に分かる。

## 実装ルール

### ルール 0: 単純さを先に

コードを書く前に「動き得る最も単純なものは何か」と問う。

コードを書いた後、次の観点で見直す:
- これはもっと少ない行でできないか
- これらの抽象化は複雑さに見合っているか
- スタッフエンジニアが見たら「なぜ単にこうしなかったのか」と言わないか
- 仮想の将来要件のために作っていないか。現在のタスクのために作っているか

```
単純さチェック:
✗ 1 つの通知のための middleware pipeline 付き汎用 EventBus
✓ 単純な関数呼び出し

✗ 似た 2 つのコンポーネントのための abstract factory pattern
✓ 共有ユーティリティを持つ素直な 2 つのコンポーネント

✗ 3 つのフォームのための設定駆動 form builder
✓ 3 つのフォームコンポーネント
```

似た 3 行のコードは、早すぎる抽象化よりよい。まず naive で明らかに正しい版を実装する。正しさがテストで証明された後だけ最適化する。

### ルール 0.5: スコープ規律

タスクが必要とするものだけを触る。

してはならない:
- 変更箇所の隣にあるコードを「きれいにする」
- 変更していないファイルの import をリファクタリングする
- 完全には理解していないコメントを削除する
- 仕様にない機能を「便利そうだから」と追加する
- 読んでいるだけのファイルの構文を現代化する

タスク範囲外に改善すべきものを見つけたら、修正せず記録する:

```
気づいたが触らないもの:
- src/utils/format.ts に未使用 import がある（このタスクとは無関係）
- auth middleware のエラーメッセージは改善できる（別タスク）
→ これらのタスクを作りますか？
```

### ルール 1: 一度に 1 つ

各増分は 1 つの論理的なことだけを変える。関心ごとを混ぜない。

**悪い例:** 新コンポーネント追加、既存コンポーネントのリファクタ、ビルド設定更新を 1 コミットで行う。

**良い例:** 3 つの別コミット。各変更につき 1 つ。

### ルール 2: コンパイル可能に保つ

各増分後、プロジェクトはビルドでき、既存テストは通らなければならない。スライス間でコードベースを壊れた状態にしない。

### ルール 3: 未完成機能には feature flag

機能がユーザー向けに準備できていないが、増分をマージする必要がある場合:

```typescript
// 作業中機能用の feature flag
const ENABLE_TASK_SHARING = process.env.FEATURE_TASK_SHARING === 'true';

if (ENABLE_TASK_SHARING) {
  // 新しい共有 UI
}
```

これにより、未完成の作業を露出させずに小さな増分を main ブランチへマージできる。

### ルール 4: 安全な既定値

新しいコードは、安全で保守的な振る舞いを既定にする:

```typescript
// 安全: 既定で無効、opt-in
export function createTask(data: TaskInput, options?: { notify?: boolean }) {
  const shouldNotify = options?.notify ?? false;
  // ...
}
```

### ルール 5: ロールバックしやすくする

各増分は独立して revert 可能であるべき:

- 追加的な変更（新ファイル、新関数）は戻しやすい
- 既存コードへの変更は最小限かつ焦点を絞る
- データベースマイグレーションには対応する rollback migration を用意する
- 1 コミットで何かを削除し、その同じコミットで置き換えることを避ける。分ける

## エージェントとの作業

エージェントへ段階的実装を指示する場合:

```
計画の Task 3 を実装しましょう。

まずはデータベーススキーマ変更と API エンドポイントだけから始めてください。
UI はまだ触らないでください。次の増分で行います。

実装後、`npm test` と `npm run build` を実行し、
壊れていないことを検証してください。
```

各増分で何がスコープ内で、何がスコープ外かを明示する。

## 増分チェックリスト

各増分後に検証する:

- [ ] 変更は 1 つのことを行い、それを完結して行っている
- [ ] 既存テストがすべて引き続き通る（`npm test`）
- [ ] ビルドが成功する（`npm run build`）
- [ ] 型チェックが通る（`npx tsc --noEmit`）
- [ ] lint が通る（`npm run lint`）
- [ ] 新しい機能が期待どおり動く
- [ ] 変更が説明的なメッセージでコミットされている

**注意:** 影響を受ける可能性がある変更後に、それぞれの検証コマンドを実行する。成功後、コードが変わっていないなら同じコマンドを繰り返さない。未変更のコードで再実行しても新しい情報は増えない。

## よくある正当化

| 正当化 | 現実 |
|--------|------|
| 「最後に全部テストする」 | バグは複利で増える。Slice 1 のバグは Slice 2-5 を間違ったものにする。各スライスをテストする。 |
| 「一気にやったほうが速い」 | 何かが壊れ、500 行の変更のどこが原因か分からなくなるまでは速く *感じる*。 |
| 「この変更は小さすぎて別コミットにするほどではない」 | 小さなコミットは無料である。大きなコミットはバグを隠し、ロールバックを痛くする。 |
| 「feature flag は後で追加する」 | 機能が未完成なら、ユーザーから見えてはいけない。今 flag を追加する。 |
| 「このリファクタは小さいから含めてよい」 | 機能と混ざったリファクタは、レビューもデバッグも難しくする。分ける。 |
| 「念のためビルドコマンドをもう一度実行する」 | 成功後に同じコマンドを繰り返しても、コードが変わっていない限り何も増えない。安心のためではなく、その後の編集後に再実行する。 |

## 危険信号

- テストを実行せずに 100 行を超えるコードを書く
- 1 つの増分に複数の無関係な変更がある
- 「これもついでにすぐ追加する」というスコープ拡大
- 速く進むために test/verify ステップを省く
- 増分間でビルドまたはテストが壊れている
- 大きな未コミット変更が蓄積している
- 3 つ目のユースケースが要求する前に抽象化を作る
- 「ついでに」とタスク範囲外のファイルを触る
- 1 回限りの操作のために新しいユーティリティファイルを作る
- 間にコード変更を挟まず、同じ build/test コマンドを連続で 2 回実行する

## 検証

タスクのすべての増分を完了した後に確認する:

- [ ] 各増分が個別にテストされ、コミットされている
- [ ] テストスイート全体が通る
- [ ] ビルドがきれいである
- [ ] 機能が仕様どおり end-to-end で動く
- [ ] 未コミット変更が残っていない
