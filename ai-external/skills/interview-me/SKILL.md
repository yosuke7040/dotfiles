# インタビュー

## 概要

人が依頼するものと、本当に欲しいものは違うことが多い。「ダッシュボードが欲しい」と言うのは、それが頼み方として自然だからであって、ダッシュボードが問題を解くからとは限らない。「もっと速くして」と言っても、達成すべき数値がないこともある。

このずれを見つける一番安いタイミングは、計画・仕様・コードがまだ何もない段階である。作り始めた後は方向転換に実コストがかかり、ユーザーも間違ったものを「これで十分」と合理化しやすい。ずれはそのまま固定される。

このスキルは、コストが発生する前にずれを閉じる。他の定義フェーズのスキルは、欲しいものが大まかに分かっている前提で動く。`idea-refine` はアイデアからバリエーションを出し、`spec-driven-development` は要件を書き下し、`doubt-driven-development` は計画を作った後でストレステストする。`interview-me` はそれらの前段で、ユーザーが言う前に次に何を言いそうか予測できるようになるまで、自分の最善の推測を添えて一つずつ質問する。

## 使う場面

次の場合に使う。

- 依頼に少なくとも一つ欠けている: **誰**のためか、**なぜ**欲しいのか、**成功**は何か、縛っている**制約**は何か
- 依頼が具体ではなく定型的（「X を作って」「もっと速くして」）で、推測なしでは定型の中身を解けない
- 表明していない前提から始めたくなっている
- 単純さと柔軟性、コストと速度など、妥当な価値同士が緊張しているのに、ユーザーがどちらを最適化するか言っていない
- ユーザーが明示的に「interview me」「grill me」「始める前に本当にこれでよいか確認して」「考えをストレステストして」と呼び出している

**使わない場面:**

- 依頼が曖昧でなく自己完結している（「この変数名を変えて」「この typo を直して」）
- ユーザーが検証より速度を明示的に優先している
- 純粋な情報依頼（「X はどう動く？」「このコードは何をしている？」）
- 機械的な操作（rename、format、file move）
- すでに 95% 以上の確信がある。そう思い込む前に下の停止条件を読み直す

## 読み込み制約

このスキルには応答可能なユーザーが必要。CI pipeline、scheduled run、`/loop`、自律ループのような非対話コンテキストでは **呼び出さない**。その状況で依頼が未指定なら、推測するのではなくブロッカーとしてユーザーに伝える。

## プロセス

### Step 1: 確信度付きで仮説を書く

質問する前に、ユーザーが欲しいものについて現時点の最善の読みを **一文**で書き、正直な確信度（0-100%）を添える。

```
仮説: スタンドアップで「今どうなっているか？」に答える手段が欲しくて、「ダッシュボード」が自然に浮かんだのだと思います。
確信度: 約 30% — 誰向けか、この文脈での「指標」が何か、成功の見え方が未確認です
```

数値は誠実さを強制する。高い数値を書いたのに、次に聞く三つの質問へのユーザーの反応を実際には予測できないなら、その数値は間違っている。防御できる確信度から始める。

確信度が約 70% 未満なら、同じ行に短い理由を添える。何が未解決または欠落しているかを書く。これにより、インタビューで何を明らかにする必要があるかがユーザーに見え、数値が曖昧なシグナルになるのを防げる。

### Step 2: 推測を添えて一度に一つだけ質問する

形式:

```
質問: <焦点を絞った一つの質問>
推測: <その答えについての仮説と、その仮説に至った理由>
```

次の質問へ進む前に、ユーザーの反応を待つ。

**まとめて聞かない理由:**

- 推測をリストに埋めると、ユーザーが反応できない
- まとめ聞きは流し読みと表面的な答えを誘発する
- 三つ目の質問は、一つ目の答えで変わることが多い。先に全部聞くと間違った枠組みを固定する
- ユーザーが丁寧に考える力は有限。一問ずつ使う

**推測を添える理由:**

- 人は白紙から答えるより、間違った推測に反応する方が速い
- 自分の仮説を外部化し、外れていることを見せられるため、誠実さが保てる
- インタビューの目的である「こちらの前提」を表面化できる

ここでのリスクは、礼儀正しいユーザーが合わせて同意してしまうこと。明確に間違ってよい姿勢を見せ、時々はユーザーが押し返しそうな方向にあえて推測することで緩和する。

### Step 3: 「欲しいもの」と「欲しいと言うべきもの」を聞き分ける

最も危険なのは、ユーザーが実際に欲しいものではなく、思慮深い答えらしく聞こえることを言う回答である。次に注意する。

- 具体性なしに best practice 風の言葉へ寄る回答（「スケールできるように」「clean architecture」）
- 定型へ委ねる回答（「多くのアプリと同じ」「標準的なやり方」）
- 「たぶん...すべき」「...することになっていると思う」「良いエンジニアリングでは...」のような表現
- 「modern」「scalable」「robust」のような buzzword が、具体的な成果の代わりに目標になっている

これが聞こえたら、次の質問をする。

> *「誰かに正当化しなくていいなら、本当は何が欲しいですか？」*

この一問だけで、それまでの五問以上の仕事をすることがある。

### Step 4: ユーザー自身の言葉で意図を言い直す

確信度が高くなったら、現時点でユーザーが欲しいと思っているものを書き返す。短く（5-8 行）、できるだけユーザーの言葉を使い、行ごとに確認または修正できる形にする。

```
今の理解はこうです。

- 成果:       <一行>
- ユーザー:   <一行 — 誰が得をするか>
- なぜ今:     <一行 — 何が変わったか>
- 成功:       <一行 — どうなれば効いたと分かるか>
- 制約:       <一行 — 実際に縛っている制限>
- 範囲外:     <一行 — 明示的にやらないこと>

はい / いいえ / 修正したい点はありますか？
```

「範囲外」は必須。ずれの半分は、**何を作らないか**についての沈黙した不一致から生まれる。

### Step 5: 明示的な yes を確認する。「任せます」では足りない

ゲートは明示的な「はい」。次は **はいではない**。

- 「任せます」 → ユーザー側にも 95% の確信がないという委譲。二つの具体案を選択肢として出し、聞き直す
- 「いいと思います」 → 曖昧。「修正したい点はありますか？」と聞く。沈黙は確認ではない
- 「では進めて」 → 賛同ではなく丁寧な終了であることが多い。同じフォローをする
- 沈黙の後の「じゃあ始めましょう」 → 収束ではなくインタビューを諦めた可能性がある。止めて、見落としがないか聞く

修正が入ったら取り込み、再度言い直す。明示的な yes が得られるまで繰り返す。

### 95% 確信の停止条件

次に yes と答えられるなら完了。

> *次に聞く三つの質問へのユーザーの反応を予測できるか？*

yes なら共有理解がある。インタビューを止め、言い直しを出す。no なら未完了。次の質問をする。

これは感覚ではなく確認可能なテストである。ただし下限もある。何往復してもまだ予測できないなら、それは依頼についての情報であって、さらに粘る理由ではない。止めてユーザーへ伝える。「X 個質問しましたが、まだ反応を予測できません。根本的な情報が欠けています。少し戻りますか？」

## 出力

このスキルの出力は **確認済みの意図文**。Step 4 の言い直しと Step 5 の明示的な yes が成果物である。仕様、計画、タスク一覧は下流にあり、このスキルが作った意図を消費する。

複数セッションのプロジェクトや別の協力者への引き継ぎなど、ユーザーが意図の永続化を望む場合は `docs/intent/[topic].md` への保存を提案する。保存するのは確認を得てからだけ。

## 例

短い before / after。

**`interview-me` なし:**

```
User:  指標を見るダッシュボードを作って
Agent: <chart library と layout の提案を始める>
```

agent は、誰のためか、どの指標か、「自分たち向け」が何を意味するか、成功が何かを仮定している。どれも明示されていない。すべての仮定は、後でユーザーが「そういう意味ではなかった」と言える場所になる。

**`interview-me` あり:**

```
User: 指標を見るダッシュボードを作って

Agent:
仮説: スタンドアップで「今どうなっているか？」に答える手段が欲しくて、「ダッシュボード」が自然に浮かんだのだと思います。
確信度: 約 30% — 誰向けか、この文脈での「指標」が何か、成功の見え方が未確認です

質問: 「今どうなっているか？」と言うとき、聞いているのはあなただけですか、スタンドアップの開発チームですか、それとも上位層（manager、exec、board など）ですか？
推測: 開発チームのスタンドアップだと思っています。「自分たち」は通常その範囲を指し、この質問が出る場も standup であることが多いからです。exec 向けなら指標も見せ方もかなり変わります。

User: 実は自分用です。どの実験が走っていて、初期シグナルがどう見えているかを追えなくなっています。

Agent:
更新した理解: チーム向け dashboard ではなく、個人用の実験トラッカーです。確信度: 約 60% — まだ「初期シグナル」の意味と、完了の見え方が未確認です。

質問: 足りないのは、どの実験が存在するか分からないことですか、それとも結果を一か所で見られないことですか？
推測: 後者だと思っています。実験一覧はどこかにあるが、結果が五つの tool に散っていて、手で突き合わせるのに疲れているのではないかと見ています。

User: 前者です。実験一覧自体がありません。いろいろな docs に散っています。

Agent: <続ける>
```

二つの質問で、実際の依頼が「dashboard」ではないと分かった。「list」である。成果物も scope も作業も違う。dashboard は間違いだった。

## 他スキルとの関係

- **`idea-refine`**: 下流。確認済みの意図が「X は欲しいが scope が分からない」なら、明示された意図を使って `idea-refine` に渡し、バリエーションを出す。
- **`spec-driven-development`**: 下流。確認済みの意図が具体的（「Y users のために Z success criteria を持つ X が欲しい」）なら、`spec-driven-development` に渡して書き下す。
- **`planning-and-task-breakdown`**: このスキルの二段下流（spec の後）。
- **`doubt-driven-development`**: 時系列の反対側。`interview-me` は意思決定前の意図抽出、`doubt-driven-development` は意思決定後の成果物レビュー。どちらも乖離を捕まえるが、捕まえる時点が違う。
- **`source-driven-development`**: 直交する。`interview-me` はユーザーが何を欲しいかを明らかにし、SDD は framework の事実を確認する。競合しない。

## よくある正当化

| 正当化 | 現実 |
|---|---|
| 「依頼は十分明確」 | 今すぐユーザーの望む成果を一文で書けないなら明確ではない。判断前に Step 1 を実行する。 |
| 「質問しすぎると時間を無駄にする」 | 的を絞った 4-6 問で失う時間は小さい。間違ったものを作って失う時間は大きく、そのコストを負うのはユーザーである。 |
| 「作りながら分かる」 | コードが存在した後の切り替えコストは今の 10 倍。実装中の発見は手戻りである。 |
| 「『任せます』と言われたから決めればよい」 | 「任せます」は委譲であって決定ではない。二つの具体案を選択肢として聞き直す。 |
| 「いくつか選択肢を出して選んでもらうべき」 | 選択肢は、ユーザーが欲しいものを知っていて trade-off を選ぶ段階で効く。まだ欲しいものが分かっていない。選択肢の列挙は探索を広げ、質問は探索を狭める。 |
| 「推測を添えると誘導になる」 | 誘導することに価値がある。白紙から生成するより反応する方が速い。リスクは誘導ではなく迎合なので、間違ってよい姿勢を明示して緩和する。 |
| 「十分話したから分かった」 | テストする。次の三つの質問への反応を予測できるか？できないなら、まだ分かっていない。 |
| 「ユーザーが yes と言ったから終わり」 | 曖昧な言い直しや開いた「よさそう」の後の yes は空洞である。具体的に言い直し、再確認する。 |

## 危険信号

- 一つのメッセージに三つ以上の質問がある。これはインタビューではなくまとめ聞きである
- 仮説を添えない質問。これは commit ではなく survey である
- 「任せます」を最終回答として受け入れる
- ユーザーが言い直しを明示的に確認する前に、仕様・計画・タスク一覧を出す
- 「best practice は何か？」という形で聞き、「本当は何が欲しいか？」を聞かない
- ユーザーが「scalable」「clean」「modern」のような洗練シグナルの答えを出し、それが本当に欲しいものか掘らずに受け入れる
- 三往復以上しても確信度が目に見えて上がらない。質問が間違っているので、一歩戻って枠組みを変える
- 約 70% 未満の確信度に理由がない。何が欠けているか分からなければ、ユーザーはずれを閉じられない
- ユーザー確認前に意図 doc を保存する。doc 自体が、ユーザーが与えていない yes を暗示してしまう
- 言い直しから「範囲外」を省く。非目標についての沈黙した不一致は、ずれの半分である

## 検証

`interview-me` 適用後:

- [ ] 初回ターンで明示的な仮説と確信度を述べた
- [ ] 約 70% 未満の確信度には、未解決または欠落しているものを一行で添えた
- [ ] 質問は一つずつ行い、それぞれに agent の推測を添えた
- [ ] ユーザーが洗練シグナルまたは定型シグナルの回答をした場合、「正当化しなくてよいなら本当は何が欲しいか？」の probe を少なくとも一度行った
- [ ] 具体的な言い直し（成果 / ユーザー / なぜ今 / 成功 / 制約 / 範囲外）をユーザーに返した
- [ ] ユーザーが明示的な yes で言い直しを確認した（「任せます」「よさそう」、沈黙ではない）
- [ ] 停止時点で、次に聞く三つの質問への反応を予測できた
- [ ] 下流スキル（`idea-refine`、`spec-driven-development`）への引き継ぎは、元の未指定依頼ではなく確認済み意図に基づいていた
