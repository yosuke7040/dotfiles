# コードレビューと品質

## 概要

品質ゲートを持つ多面的なコードレビュー。すべての変更はマージ前にレビューする。例外はない。レビューは、正しさ、読みやすさ、アーキテクチャ、セキュリティ、性能の 5 軸を扱う。

**承認基準:** 完璧でなくても、全体のコード健全性を明確に改善する変更は承認する。完璧なコードは存在しない。目標は継続的改善である。自分なら同じように書かなかったという理由で変更をブロックしない。コードベースを改善し、プロジェクトの慣習に従っているなら承認する。

## 使う場面

- PR または変更をマージする前
- 機能実装完了後
- 別エージェントまたはモデルが生成したコードを評価する必要がある
- 既存コードをリファクタリングした後
- バグ修正後（修正と regression test の両方をレビューする）

## 5 軸レビュー

すべてのレビューは、次の観点でコードを評価する。

### 1. 正しさ

コードは主張どおりのことをしているか。

- 仕様またはタスク要件と一致しているか
- エッジケース（null、空、境界値）は処理されているか
- エラー経路は処理されているか（正常系だけでないか）
- すべてのテストが通るか。テストは本当に正しいものをテストしているか
- off-by-one エラー、競合状態、状態の不整合はないか

### 2. 読みやすさと単純さ

別のエンジニア、またはエージェントが、作者の説明なしでこのコードを理解できるか。

- 名前は説明的で、プロジェクトの慣習と一貫しているか（文脈なしの `temp`、`data`、`result` は避ける）
- 制御フローは素直か（ネストした三項演算子、深い callback を避ける）
- コードは論理的に整理されているか（関連コードがまとまり、モジュール境界が明確か）
- 単純化すべき「気の利いた」技巧はないか
- **これはもっと少ない行でできるか。** 100 行で足りるところに 1000 行あるなら失敗である
- **抽象化は複雑さに見合っているか。** 3 つ目のユースケースまで一般化しない
- 自明でない意図を明確にするコメントは役立つか（ただし自明なコードにはコメントしない）
- dead code の痕跡はないか。no-op 変数（`_unused`）、後方互換 shim、`// removed` コメントなど

### 3. アーキテクチャ

変更はシステム設計に合っているか。

- 既存パターンに従っているか。それとも新しいパターンを導入しているか。新しい場合は正当化されているか
- きれいなモジュール境界を保っているか
- 共有すべきコード重複はないか
- 依存関係は正しい方向に流れているか（循環依存はないか）
- 抽象化の粒度は適切か（過剰設計でも密結合でもないか）

### 4. セキュリティ

詳細なセキュリティ指針は `security-and-hardening` を参照する。変更が脆弱性を導入していないか。

- ユーザー入力は検証、サニタイズされているか
- シークレットはコード、ログ、バージョン管理から除外されているか
- 必要な箇所で認証と認可が確認されているか
- SQL クエリはパラメータ化されているか（文字列連結なし）
- XSS を防ぐために出力がエンコードされているか
- 依存関係は信頼済みの情報源から来ており、既知の脆弱性がないか
- 外部情報源（API、logs、user content、config files）のデータを信頼できないものとして扱っているか
- 外部データフローは、ロジックまたはレンダリングで使う前にシステム境界で検証されているか

### 5. 性能

詳細なプロファイリングと最適化は `performance-optimization` を参照する。変更が性能問題を導入していないか。

- N+1 query パターンはないか
- 上限のないループや制約のないデータ取得はないか
- async にすべき同期処理はないか
- UI コンポーネントに不要な再レンダーはないか
- list endpoint に pagination 漏れはないか
- hot path で大きなオブジェクトを作っていないか

## 変更サイズ

小さく焦点の合った変更はレビューしやすく、マージしやすく、デプロイも安全である。目標サイズ:

```
変更約 100 行   → 良い。1 回でレビューできる。
変更約 300 行   → 単一の論理変更なら許容。
変更約 1000 行  → 大きすぎる。分割する。
```

**「1 つの変更」とは:** 1 つのことに対処する自己完結した修正で、関連テストを含み、提出後もシステムが動作するもの。機能全体ではなく、機能の一部である。

**変更が大きすぎる場合の分割戦略:**

| 戦略 | 方法 | 使う場面 |
|------|------|----------|
| **Stack** | 小さな変更を出し、その上に次を積む | 逐次依存 |
| **ファイルグループ別** | 異なるレビュアーが必要なグループごとに分ける | 横断的関心事 |
| **水平** | 先に共有コードや stub を作り、その後 consumer を作る | レイヤー型アーキテクチャ |
| **垂直** | 機能をより小さな full-stack slice へ分ける | 機能作業 |

**大きな変更が許容される場合:** 完全なファイル削除、およびレビュアーが各行ではなく意図だけを確認すればよい自動リファクタリング。

**リファクタリングと機能作業は分ける。** 既存コードをリファクタリングし、新しい振る舞いも追加する変更は 2 つの変更である。別々に提出する。小さな cleanup（変数名変更）はレビュアー判断で含めてよい。

## 変更説明

すべての変更には、バージョン管理履歴の中で単独で読める説明が必要である。

**1 行目:** 短く、命令形で、単独で意味が分かるもの。「Deleting the FizzBuzz RPC」ではなく「Delete the FizzBuzz RPC」。履歴を検索する人が diff を読まなくても変更を理解できるだけの情報を含める。

**本文:** 何がなぜ変わるのか。コード自体からは見えない文脈、判断、理由を含める。関連する bug 番号、benchmark 結果、設計文書へリンクする。アプローチに弱点がある場合は認める。

**アンチパターン:** 「Fix bug」「Fix build」「Add patch」「Moving code from A to B」「Phase 1」「Add convenience functions」。

## レビュープロセス

### ステップ 1: 文脈を理解する

コードを見る前に意図を理解する:

```
- この変更は何を達成しようとしているか
- どの仕様またはタスクを実装しているか
- 期待される振る舞いの変化は何か
```

### ステップ 2: 最初にテストをレビューする

テストは意図とカバレッジを明らかにする:

```
- 変更に対するテストはあるか
- テストは実装詳細ではなく振る舞いをテストしているか
- エッジケースはカバーされているか
- テスト名は説明的か
- コード変更時に回帰を捕まえられるテストか
```

### ステップ 3: 実装をレビューする

5 軸を意識してコードを追う:

```
変更された各ファイルについて:
1. 正しさ: このコードはテストが求めることをしているか
2. 読みやすさ: 助けなしに理解できるか
3. アーキテクチャ: システムに合っているか
4. セキュリティ: 脆弱性はないか
5. 性能: ボトルネックはないか
```

### ステップ 4: 指摘を分類する

すべてのコメントに重大度ラベルを付け、作者が必須と任意を区別できるようにする:

| Prefix | 意味 | 作者の対応 |
|--------|------|------------|
| *(prefix なし)* | 必須変更 | マージ前に対応が必要 |
| **Critical:** | マージをブロック | セキュリティ脆弱性、データ損失、壊れた機能 |
| **Nit:** | 軽微、任意 | 作者は無視してよい。整形、スタイル嗜好 |
| **Optional:** / **Consider:** | 提案 | 検討に値するが必須ではない |
| **FYI** | 情報のみ | 対応不要。将来の文脈 |

これにより、作者がすべてのフィードバックを必須と受け取り、任意提案に時間を浪費することを防ぐ。

### ステップ 5: 検証内容を検証する

作者の検証ストーリーを確認する:

```
- どのテストが実行されたか
- ビルドは通ったか
- 手動でテストされたか
- UI 変更にスクリーンショットはあるか
- before/after 比較はあるか
```

## 複数モデルレビューのパターン

異なるレビュー視点には異なるモデルを使う:

```
Model A がコードを書く
    │
    ▼
Model B が正しさとアーキテクチャをレビューする
    │
    ▼
Model A がフィードバックに対応する
    │
    ▼
人間が最終判断する
```

これは単一モデルが見逃す問題を捕まえる。異なるモデルは異なる盲点を持つ。

**レビューエージェント向けプロンプト例:**
```
このコード変更を、正しさ、セキュリティ、プロジェクト慣習への準拠でレビューしてください。
仕様では [X] とされています。この変更は [Y] であるべきです。
問題があれば Critical、Important、Suggestion として分類してください。
```

## dead code 衛生

リファクタリングまたは実装変更後は、孤立したコードを確認する:

1. 到達不能または未使用になったコードを特定する
2. 明示的に列挙する
3. **削除前に確認する:** 「これらの未使用要素を削除してよいですか: [list]?」

dead code を放置しない。将来の読者やエージェントを混乱させる。ただし、確信がないものを黙って削除してはならない。迷ったら聞く。

```
特定した dead code:
- src/utils/date.ts の formatLegacyDate()。formatDate() に置き換え済み
- src/components/ の OldTaskCard component。TaskCard に置き換え済み
- src/config.ts の LEGACY_API_URL constant。参照が残っていない
→ これらを削除してよいですか？
```

## レビュー速度

遅いレビューはチーム全体をブロックする。レビューへのコンテキストスイッチのコストは、他者に課す待ち時間のコストより小さい。

- **1 営業日以内に応答する**。これは最大値であり、目標ではない
- **理想的な cadence:** 深い集中実装中でない限り、レビュー依頼が来たらすぐ応答する。典型的な変更は 1 日で複数レビューラウンドを完了するべきである
- **最終承認の速さより、個々の応答の速さを優先する。** 複数ラウンドが必要でも、素早いフィードバックは不満を減らす
- **大きな変更:** 巨大な changeset をレビューするのではなく、作者へ分割を依頼する

## 意見の不一致の扱い

レビュー上の対立を解決するときは、この階層を適用する:

1. **技術的事実とデータ** は意見や好みより優先する
2. **スタイルガイド** はスタイル問題における絶対的な権威である
3. **ソフトウェア設計** は個人的好みではなくエンジニアリング原則で評価する
4. **コードベース一貫性** は、全体の健全性を悪化させないなら受け入れられる

**「後で片付ける」を受け入れない。** 経験上、先送りされた cleanup はほとんど起きない。本物の緊急事態でない限り、提出前に cleanup を求める。周辺問題をこの変更で対応できない場合は、自己アサイン付きの bug を起票させる。

## レビューでの誠実さ

自分、別エージェント、人間のいずれが書いたコードをレビューする場合でも:

- **rubber-stamp しない。** レビューの証拠がない "LGTM" は誰の役にも立たない。
- **実問題を弱めて言わない。** 本番で当たるバグなのに「小さな懸念かもしれない」と言うのは不誠実である。
- **可能なら問題を定量化する。** 「この N+1 query は一覧の各 item ごとに約 50ms 増やす」のほうが「遅いかもしれない」よりよい。
- **明確な問題があるアプローチには反対する。** レビューにおける迎合は失敗モードである。実装に問題があるなら直接述べ、代替案を示す。
- **override は穏当に受け入れる。** 作者が完全な文脈を持ち、同意しない場合はその判断を尊重する。人ではなくコードにコメントする。個人批判はコード自体へ焦点を戻す。

## 依存関係の規律

コードレビューには依存関係レビューも含まれる。

**依存関係を追加する前に:**
1. 既存 stack で解けないか（多くの場合は解ける）
2. 依存関係はどれほど大きいか（bundle impact を確認）
3. 活発に保守されているか（last commit、open issues を確認）
4. 既知の脆弱性があるか（`npm audit`）
5. ライセンスは何か（プロジェクトと互換でなければならない）

**ルール:** 新しい依存関係より、標準ライブラリと既存ユーティリティを優先する。すべての依存関係は負債である。

## レビューチェックリスト

```markdown
## レビュー: [PR/変更タイトル]

### 背景
- [ ] この変更が何をなぜ行うかを理解している

### 正しさ
- [ ] 変更は仕様/タスク要件に合っている
- [ ] エッジケースが処理されている
- [ ] エラー経路が処理されている
- [ ] テストが変更を十分にカバーしている

### 読みやすさ
- [ ] 名前が明確で一貫している
- [ ] ロジックが素直である
- [ ] 不要な複雑さがない

### アーキテクチャ
- [ ] 既存パターンに従っている
- [ ] 不要な coupling や dependencies がない
- [ ] 抽象化の粒度が適切である

### セキュリティ
- [ ] コード内にシークレットがない
- [ ] 入力が境界で検証されている
- [ ] injection 脆弱性がない
- [ ] auth checks がある
- [ ] 外部データソースを信頼できないものとして扱っている

### 性能
- [ ] N+1 パターンがない
- [ ] 上限のない操作がない
- [ ] list endpoint に pagination がある

### 検証
- [ ] テストが通る
- [ ] ビルドが成功する
- [ ] 手動検証が行われている（該当する場合）

### 判定
- [ ] **Approve**: マージ準備完了
- [ ] **Request changes**: 対応が必要
```

## よくある正当化

| 正当化 | 現実 |
|--------|------|
| 「動くから十分」 | 読みにくい、安全でない、またはアーキテクチャ的に間違った動くコードは、複利で増える負債を作る。 |
| 「自分が書いたから正しいと分かる」 | 作者は自分の仮定に盲目である。すべての変更は別の目から利益を得る。 |
| 「後で片付ける」 | 後では来ない。レビューは品質ゲートである。マージ後ではなく前に cleanup を求める。 |
| 「AI 生成コードはたぶん大丈夫」 | AI コードにはより多くの精査が必要であり、少なくてよいわけではない。間違っていても自信満々でそれらしい。 |
| 「テストが通るから良い」 | テストは必要だが十分ではない。アーキテクチャ問題、セキュリティ問題、読みやすさの問題は捕まえない。 |

## 危険信号

- レビューなしで PR がマージされる
- テストが通るかだけを確認するレビュー（他の軸を無視）
- 実際にレビューした証拠のない "LGTM"
- セキュリティに敏感な変更に、セキュリティ重視レビューがない
- 「適切にレビューするには大きすぎる」大 PR（分割する）
- バグ修正 PR に regression test がない
- レビューコメントに重大度ラベルがなく、必須と任意が分からない
- 「後で直す」を受け入れる。後では起きない

## 検証

レビュー完了後に確認する:

- [ ] Critical issues がすべて解決されている
- [ ] Important issues がすべて解決されているか、根拠付きで明示的に defer されている
- [ ] テストが通る
- [ ] ビルドが成功する
- [ ] 検証ストーリーが文書化されている（何が変わり、どう検証されたか）
