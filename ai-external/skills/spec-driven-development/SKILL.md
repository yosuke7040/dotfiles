# 仕様駆動開発

## 概要

コードを書く前に、構造化された仕様を書く。仕様は、あなたと人間のエンジニアの間で共有される信頼できる情報源である。何を作るのか、なぜ作るのか、どうなれば完了と分かるのかを定義する。仕様なしのコードは推測である。

## 使う場面

- 新しいプロジェクトまたは機能を始める
- 要件が曖昧または不完全である
- 変更が複数のファイルまたはモジュールにまたがる
- アーキテクチャ上の判断をしようとしている
- 実装に 30 分以上かかりそうである

**使わない場面:** 1 行修正、typo 修正、または要件が曖昧さなく自己完結している変更。

## ゲート付きワークフロー

仕様駆動開発には 4 つのフェーズがある。現在のフェーズが検証されるまで、次のフェーズへ進んではならない。

```
SPECIFY ──→ PLAN ──→ TASKS ──→ IMPLEMENT
   │          │        │          │
   ▼          ▼        ▼          ▼
 Human      Human    Human      Human
 reviews    reviews  reviews    reviews
```

### フェーズ 1: 仕様化

高レベルのビジョンから始める。要件が具体的になるまで、人間に明確化の質問をする。

**仮定をすぐ表面化する。** 仕様本文を書く前に、何を仮定しているかを列挙する:

```
こちらで置いている仮定:
1. これは Web アプリケーションである（ネイティブモバイルではない）
2. 認証はセッションベースの Cookie を使う（JWT ではない）
3. データベースは PostgreSQL である（既存の Prisma schema に基づく）
4. 対象はモダンブラウザのみである（IE11 は対象外）
→ 違っていれば今指摘してください。なければこの前提で進めます。
```

曖昧な要件を黙って埋めてはならない。仕様の目的は、コードを書く *前* に誤解を表面化することである。仮定は最も危険な誤解の形である。

**次の 6 つの中核領域を含む仕様文書を書く:**

1. **目的**: 何を、なぜ作るのか。ユーザーは誰か。成功とは何か。

2. **コマンド**: ツール名だけでなく、フラグを含む完全に実行可能なコマンド。
   ```
   Build: npm run build
   Test: npm test -- --coverage
   Lint: npm run lint --fix
   Dev: npm run dev
   ```

3. **プロジェクト構造**: ソースコード、テスト、ドキュメントをどこに置くか。
   ```
   src/           → アプリケーションのソースコード
   src/components → React コンポーネント
   src/lib        → 共有ユーティリティ
   tests/         → ユニットテストとインテグレーションテスト
   e2e/           → エンドツーエンドテスト
   docs/          → ドキュメント
   ```

4. **コードスタイル**: スタイルを説明する 3 段落より、実際のコード断片 1 つのほうが強い。命名規則、整形ルール、望ましい出力例を含める。

5. **テスト戦略**: 使うフレームワーク、テストの場所、カバレッジ期待値、関心ごとごとにどのテストレベルを使うか。

6. **境界**: 3 層のシステム:
   - **常に行う:** コミット前にテストを実行する、命名規則に従う、入力を検証する
   - **先に確認する:** データベーススキーマ変更、依存関係追加、CI 設定変更
   - **決して行わない:** シークレットをコミットする、vendor ディレクトリを編集する、承認なしに失敗テストを削除する

**仕様テンプレート:**

```markdown
# 仕様: [プロジェクト名/機能名]

## 目的
[何をなぜ作るか。ユーザーストーリーまたは受け入れ条件。]

## 技術スタック
[フレームワーク、言語、主要依存関係とバージョン]

## コマンド
[build、test、lint、dev の完全なコマンド]

## プロジェクト構造
[説明付きのディレクトリ構成]

## コードスタイル
[コード例 + 主要な慣習]

## テスト戦略
[フレームワーク、テスト場所、カバレッジ要件、テストレベル]

## 境界
- Always: [...]
- Ask first: [...]
- Never: [...]

## 成功条件
[完了を判断する方法。具体的でテスト可能な条件]

## 未解決の質問
[人間の入力が必要な未解決事項]
```

**指示を成功条件に言い換える。** 曖昧な要件を受け取ったら、具体的な条件へ変換する:

```
要件: 「ダッシュボードを速くして」

言い換えた成功条件:
- 4G 接続で Dashboard LCP < 2.5s
- 初期データ読み込みが < 500ms で完了
- 読み込み中にレイアウトシフトがない（CLS < 0.1）
→ この目標で合っていますか？
```

これにより、「速い」が何を意味するかを推測するのではなく、明確な目標へ向けて反復し、再試行し、問題解決できる。

### フェーズ 2: 計画

検証済みの仕様をもとに、技術的な実装計画を作る:

1. 主要コンポーネントと依存関係を特定する
2. 実装順序を決める（何を先に作る必要があるか）
3. リスクと緩和策を記録する
4. 並列で作れるものと、逐次にする必要があるものを特定する
5. フェーズ間の検証チェックポイントを定義する

計画はレビュー可能でなければならない。人間が読んで「はい、その方針でよい」または「いいえ、X を変えて」と言える状態にする。

### フェーズ 3: タスク

計画を、個別に実装可能なタスクへ分解する:

- 各タスクは 1 回の集中セッションで完了できる
- 各タスクには明示的な受け入れ条件がある
- 各タスクには検証手順（テスト、ビルド、手動確認）がある
- タスクは重要そうに見える順ではなく、依存関係順に並べる
- 1 つのタスクで変更するファイルはおおむね 5 ファイル以内にする

**タスクテンプレート:**
```markdown
- [ ] Task: [説明]
  - Acceptance: [完了時に真でなければならないこと]
  - Verify: [確認方法。テストコマンド、ビルド、手動確認]
  - Files: [触るファイル]
```

### フェーズ 4: 実装

`skills/incremental-implementation/SKILL.md`（`incremental-implementation`）と `skills/test-driven-development/SKILL.md`（`test-driven-development`）に従い、タスクを 1 つずつ実行する。毎ステップで仕様全体をエージェントに流し込むのではなく、`skills/context-engineering/SKILL.md`（`context-engineering`）を使って、適切な仕様セクションとソースファイルを読み込む。

## 保存場所

仕様は作業中リポジトリの `.tmp/<作業ディレクトリ>/SPEC.md` として保存する。作業ディレクトリ命名規則:

```
.tmp/<YYYY-MM-DD>-<日本語スラッグ>[-<LINEAR-KEY>]/
```

- `<YYYY-MM-DD>`: 仕様作成日
- `<日本語スラッグ>`: 修正改修内容を 10〜20 文字程度で表す日本語名。空白・記号はハイフン置換（例: `ユーザー認証追加`）
- `<LINEAR-KEY>`: Linear issue がある場合のみ末尾に付与（例: `-ENG-123`）
- 同日同名衝突時は `-2`, `-3` をサフィックス

`.tmp/` は通常 gitignore 対象である前提で、SPEC.md は履歴に残らない作業用ドキュメントとして扱う。下流の `/plan` `/build` は同じディレクトリ配下に `PLAN.md` `TODO.md` を置く。

## 仕様を生かし続ける

仕様は一度きりの成果物ではなく、生きた文書である:

- **判断が変わったら更新する**: データモデル変更が必要だと分かったら、先に仕様を更新してから実装する。
- **スコープが変わったら更新する**: 追加または削除された機能は仕様に反映する。
- **作業中は手元で更新し続ける**: `.tmp/` 配下にあり gitignore されているため履歴は残らないが、作業中は信頼できる情報源として更新し続ける。PR を作る際は、必要に応じて要点を PR description や ADR に転記する。

## よくある正当化

| 正当化 | 現実 |
|--------|------|
| 「これは単純だから仕様はいらない」 | 単純なタスクに *長い* 仕様は不要だが、受け入れ条件は必要である。2 行の仕様でもよい。 |
| 「コードを書いてから仕様を書く」 | それは仕様ではなくドキュメントである。仕様の価値は、コードの *前* に明確さを強制する点にある。 |
| 「仕様を書くと遅くなる」 | 15 分の仕様は何時間もの手戻りを防ぐ。15 分の waterfall は 15 時間のデバッグよりましである。 |
| 「要件はどうせ変わる」 | だから仕様は生きた文書である。古い仕様でも、仕様がないよりはよい。 |
| 「ユーザーは自分がほしいものを分かっている」 | 明確な依頼にも暗黙の仮定がある。仕様はその仮定を表面化する。 |

## 危険信号

- 書かれた要件なしでコードを書き始める
- 「完了」の意味を明確にする前に「とりあえず作り始めますか」と聞く
- 仕様またはタスクリストにない機能を実装する
- アーキテクチャ判断を文書化せずに行う
- 「何を作るかは明らかだから」と仕様を省く

## 検証

実装へ進む前に確認する:

- [ ] 仕様が 6 つの中核領域すべてをカバーしている
- [ ] 人間が仕様をレビューし、承認している
- [ ] 成功条件が具体的でテスト可能である
- [ ] 境界（Always / Ask First / Never）が定義されている
- [ ] 仕様が `.tmp/<作業ディレクトリ>/SPEC.md` に保存されている
