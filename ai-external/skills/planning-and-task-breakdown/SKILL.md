# 計画とタスク分解

## 概要

作業を、明示的な受け入れ条件を持つ小さく検証可能なタスクへ分解する。良いタスク分解は、エージェントが確実に作業を完了するか、絡まった混乱を作るかの分かれ目になる。すべてのタスクは、1 回の集中セッションで実装、テスト、検証できる大きさにする。

## 使う場面

- 仕様があり、それを実装可能な単位へ分ける必要がある
- タスクが大きすぎる、または曖昧すぎて始めにくい
- 複数のエージェントまたはセッションへ作業を並列化する必要がある
- 人間にスコープを伝える必要がある
- 実装順序が明らかでない

**使わない場面:** スコープが明らかな単一ファイル変更、または仕様に十分定義されたタスクがすでに含まれている場合。

## 計画プロセス

### ステップ 1: 計画モードへ入る

コードを書く前に、読み取り専用モードで動く:

- 仕様と関連するコードベース箇所を読む
- 既存パターンと慣習を特定する
- コンポーネント間の依存関係を対応づける
- リスクと不明点を記録する

**計画中にコードを書いてはならない。** 出力は実装ではなく計画文書である。

### ステップ 2: 依存グラフを特定する

何が何に依存するかを対応づける:

```
Database schema
    │
    ├── API models/types
    │       │
    │       ├── API endpoints
    │       │       │
    │       │       └── Frontend API client
    │       │               │
    │       │               └── UI components
    │       │
    │       └── Validation logic
    │
    └── Seed data / migrations
```

実装順序は依存グラフの下から上へ従う。基礎を先に作る。

### ステップ 3: 垂直に分割する

データベース全体、API 全体、UI 全体を順に作るのではなく、一度に 1 つの完結した機能経路を作る。

**悪い例（水平分割）:**
```
Task 1: データベーススキーマ全体を作る
Task 2: すべての API エンドポイントを作る
Task 3: すべての UI コンポーネントを作る
Task 4: すべてを接続する
```

**良い例（垂直分割）:**
```
Task 1: ユーザーがアカウントを作成できる（登録用 schema + API + UI）
Task 2: ユーザーがログインできる（auth schema + ログイン API + UI）
Task 3: ユーザーがタスクを作成できる（task schema + 作成 API + UI）
Task 4: ユーザーがタスクリストを見られる（query + API + 一覧 UI）
```

各垂直スライスは、動作し、テスト可能な機能を提供する。

### ステップ 4: タスクを書く

各タスクは次の構造に従う:

```markdown
## タスク [N]: [短く説明的なタイトル]

**説明:** このタスクが達成することを 1 段落で説明する。

**受け入れ条件:**
- [ ] [具体的でテスト可能な条件]
- [ ] [具体的でテスト可能な条件]

**検証:**
- [ ] テスト成功: `npm test -- --grep "feature-name"`
- [ ] ビルド成功: `npm run build`
- [ ] 手動確認: [確認内容]

**依存関係:** [依存するタスク番号、または "なし"]

**触る可能性が高いファイル:**
- `src/path/to/file.ts`
- `tests/path/to/test.ts`

**想定スコープ:** [小: 1-2 files | 中: 3-5 files | 大: 5+ files]
```

### ステップ 5: 順序づけとチェックポイント

次を満たすようタスクを並べる:

1. 依存関係が満たされる（基礎を先に作る）
2. 各タスク後にシステムが動く状態で残る
3. 2 から 3 タスクごとに検証チェックポイントがある
4. 高リスクのタスクを早めに置く（早く失敗する）

明示的なチェックポイントを追加する:

```markdown
## チェックポイント: タスク 1-3 の後
- [ ] すべてのテストが通る
- [ ] アプリケーションがエラーなくビルドされる
- [ ] 中核ユーザーフローが end-to-end で動く
- [ ] 続行前に人間とレビューする
```

## タスクサイズの指針

| サイズ | ファイル数 | スコープ | 例 |
|--------|------------|----------|----|
| **XS** | 1 | 単一関数または設定変更 | バリデーションルール追加 |
| **S** | 1-2 | 1 つのコンポーネントまたはエンドポイント | 新しい API エンドポイント追加 |
| **M** | 3-5 | 1 つの機能スライス | ユーザー登録フロー |
| **L** | 5-8 | 複数コンポーネントの機能 | フィルタとページネーション付き検索 |
| **XL** | 8+ | **大きすぎる。さらに分割する** | - |

タスクが L 以上なら、より小さなタスクに分解する。エージェントは S と M のタスクで最もよく動く。

**タスクをさらに分解すべき場面:**
- 1 回の集中セッションを超える（おおむね 2 時間以上のエージェント作業）
- 受け入れ条件を 3 つ以下の箇条書きで説明できない
- 2 つ以上の独立したサブシステムに触れる（例: auth と billing）
- タスクタイトルに「and」を書きたくなる（2 つのタスクである兆候）

## 計画文書テンプレート

```markdown
# 実装計画: [機能名/プロジェクト名]

## 概要
[何を作るかの 1 段落要約]

## アーキテクチャ判断
- [主要判断 1 と根拠]
- [主要判断 2 と根拠]

## タスクリスト

### フェーズ 1: 基盤
- [ ] Task 1: ...
- [ ] Task 2: ...

### チェックポイント: 基盤
- [ ] テストが通り、ビルドがきれい

### フェーズ 2: 中核機能
- [ ] Task 3: ...
- [ ] Task 4: ...

### チェックポイント: 中核機能
- [ ] End-to-end フローが動く

### フェーズ 3: 仕上げ
- [ ] Task 5: ...
- [ ] Task 6: ...

### チェックポイント: 完了
- [ ] すべての受け入れ条件を満たす
- [ ] レビュー準備完了

## リスクと緩和策
| リスク | 影響 | 緩和策 |
|------|--------|------------|
| [リスク] | [高/中/低] | [戦略] |

## 未解決の質問
- [人間の入力が必要な質問]
```

## 並列化の機会

複数のエージェントまたはセッションが利用できる場合:

- **並列化してよい:** 独立した機能スライス、実装済み機能のテスト、ドキュメント
- **逐次にする必要がある:** データベースマイグレーション、共有状態の変更、依存チェーン
- **調整が必要:** API 契約を共有する機能（まず契約を定義し、その後に並列化する）

## よくある正当化

| 正当化 | 現実 |
|--------|------|
| 「進めながら考える」 | それで絡まった混乱と手戻りが生まれる。10 分の計画は何時間も節約する。 |
| 「タスクは明らか」 | それでも書き出す。明示的なタスクは隠れた依存関係と忘れたエッジケースを表面化する。 |
| 「計画はオーバーヘッド」 | 計画こそタスクである。計画なしの実装はただのタイピングである。 |
| 「全部頭に入れておける」 | コンテキストウィンドウは有限である。書かれた計画はセッション境界と compaction を越えて残る。 |

## 危険信号

- 書かれたタスクリストなしで実装を始める
- 受け入れ条件なしに「機能を実装する」とだけ書かれたタスク
- 計画に検証手順がない
- すべてのタスクが XL サイズ
- タスク間のチェックポイントがない
- 依存順序が考慮されていない

## 検証

実装を始める前に確認する:

- [ ] すべてのタスクに受け入れ条件がある
- [ ] すべてのタスクに検証手順がある
- [ ] タスク依存関係が特定され、正しく順序づけられている
- [ ] 1 つのタスクが触るファイルはおおむね 5 ファイル以内である
- [ ] 主要フェーズ間にチェックポイントがある
- [ ] 人間が計画をレビューし、承認している
