# エージェントペルソナ

単一の役割と単一の視点を担う専門ペルソナである。各ペルソナは Markdown ファイルであり、ハーネス（Claude Code、Cursor、Copilot など）からシステムプロンプトとして消費される。

| ペルソナ | 役割 | 適した用途 |
|----------|------|------------|
| [code-reviewer](../agents/code-reviewer.md) | シニアスタッフエンジニア | マージ前の 5 軸レビュー |
| [security-auditor](../agents/security-auditor.md) | セキュリティエンジニア | 脆弱性検出、OWASP 形式の監査 |
| [test-engineer](../agents/test-engineer.md) | QA エンジニア | テスト戦略、カバレッジ分析、Prove-It パターン |
| [web-performance-auditor](../agents/web-performance-auditor.md) | Web 性能エンジニア | Core Web Vitals 監査、読み込み/レンダリング/ネットワーク分析 |

## ペルソナ、スキル、コマンドの関係

3 つの層があり、それぞれ異なる責務を持つ:

| 層 | 何か | 例 | 構成上の役割 |
|----|------|----|--------------|
| **スキル** | 手順と終了条件を持つワークフロー | `code-review-and-quality` | *どうやるか*。ペルソナまたはコマンドの内部から起動される |
| **ペルソナ** | 視点と出力形式を持つ役割 | `code-reviewer` | *誰が見るか*。視点を採用し、レポートを作る |
| **コマンド** | ユーザー向けの入口 | `/review`, `/ship` | *いつ使うか*。ペルソナとスキルを構成する |

ユーザー、またはスラッシュコマンドがオーケストレーターである。**ペルソナは他のペルソナを呼び出さない。** スキルは、ペルソナのワークフロー内で必須の経由点である。

## それぞれを使う場面

### ペルソナの直接起動
現在の変更に対して 1 つの視点がほしく、ユーザーが判断に関与する場合に選ぶ。

- 「この PR をレビューして」→ `code-reviewer` を直接起動
- 「`auth.ts` にセキュリティ問題はある？」→ `security-auditor` を直接起動
- 「checkout フローに足りないテストは何？」→ `test-engineer` を直接起動
- 「商品ページの Core Web Vitals を監査して」→ `web-performance-auditor` を直接起動

### スラッシュコマンド（背後に単一ペルソナ）
毎回説明し直すことになる反復可能なワークフローがある場合に選ぶ。

- `/review` → プロジェクトのレビュースキルとともに `code-reviewer` を包む
- `/test` → TDD スキルとともに `test-engineer` を包む
- `/webperf` → Web アプリに対する性能重視監査として `web-performance-auditor` を包む

### スラッシュコマンド（オーケストレーター、ファンアウト）
**独立した** 調査を並列に実行でき、そのレポートを単一のエージェントが統合できる場合だけ選ぶ。

- `/ship` → `code-reviewer` + `security-auditor` + `test-engineer` へ並列にファンアウトし、そのレポートを go/no-go 判断へ統合する

このリポジトリが推奨するオーケストレーションパターンはこれだけである。完全なパターンカタログとアンチパターンは [references/orchestration-patterns.md](../references/orchestration-patterns.md) を参照する。

## 判断マトリクス

```
作業は単一成果物に対する単一視点か
├── はい → ペルソナを直接起動
└── いいえ → サブタスクは独立しているか（共有可変状態なし、順序依存なし）
              ├── はい → 並列ファンアウト付きスラッシュコマンド（例: /ship）
              └── いいえ → ユーザーが逐次スラッシュコマンドを実行（/spec → /plan → /build → /test → /review）
```

## 例: 有効なオーケストレーション

`/ship` は、このリポジトリにおける標準的なファンアウト型オーケストレーターである:

```
/ship
  ├── （並列）code-reviewer    → レビューレポート
  ├── （並列）security-auditor → 監査レポート
  └── （並列）test-engineer    → カバレッジレポート
                  ↓
        統合フェーズ（メインエージェント）
                  ↓
        go/no-go 判断 + ロールバック計画
```

これが機能する理由:
- 各サブエージェントは同じ差分を扱うが、**異なる視点** の成果物を作る
- 互いに依存していないため、本物の並列性があり、実時間を短縮できる
- それぞれ新しいコンテキストウィンドウで実行されるため、メインセッションが散らからない
- 統合ステップは小さく、完全な文脈の恩恵を受けるため、メインエージェントに残す価値がある

## 例: 無効なオーケストレーション（作らない）

「どのペルソナを呼ぶか決める」ことを役割にした `meta-orchestrator` ペルソナ:

```
/work-on-pr → meta-orchestrator
                  ↓（「これはレビューが必要」と判断）
              code-reviewer
                  ↓（戻る）
              meta-orchestrator（結果を言い換える）
                  ↓
              user
```

これが失敗する理由:
- 純粋なルーティング層で、ドメイン価値がない
- 言い換えの段を 2 つ追加し、情報損失と 2 倍のトークンコストを生む
- ユーザーはすでにレビューがほしいと分かっている。直接 `/review` を呼べばよい
- スラッシュコマンドと `AGENTS.md` の意図マッピングがすでに担う仕事を重複させる

## ペルソナのルール

1. ペルソナは、単一の出力形式を持つ単一の役割である。2 つ目の役割を追加したくなったら、2 つ目のペルソナを作る。
2. **ペルソナは他のペルソナを起動しない。** 構成はスラッシュコマンドまたはユーザーの仕事である。Claude Code では、これはプラットフォーム上の制約でもある。*"subagents cannot spawn other subagents"* であり、このルールは自動的に強制される。
3. ペルソナはスキル（*どうやるか*）を起動してよい。
4. すべてのペルソナファイルは、どこに位置づけられるかを示す "Composition" ブロックで終える。

## Claude Code との相互運用

このリポジトリのペルソナは、変更なしで Claude Code サブエージェントおよび Agent Teams のチームメイトとして動作するよう設計されている:

- **サブエージェントとして:** このプラグインが有効なとき自動発見される（パス設定は不要）。Agent ツールを `subagent_type: code-reviewer`（または `security-auditor`、`test-engineer`）で使う。`/ship` が標準例である。
- **Agent Teams のチームメイトとして**（実験的、`CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS=1` が必要）: チームメイト起動時に同じペルソナ名を参照する。ペルソナ本文は、チームメイトのシステムプロンプトに追加指示として **追記** される（置き換えではない）。そのため、ペルソナ文はリードが設定するチーム連携指示（SendMessage、タスクリストツールなど）の上に載る。

サブエージェントは結果だけをメインエージェントへ報告する。Agent Teams では、チームメイト同士が直接メッセージを送れる。レポートだけで十分な場合はサブエージェントを使う。サブエージェント同士が互いの指摘に異議を唱える必要がある場合（例: 競合仮説によるデバッグ）は Agent Teams を使う。完全な対応表は [references/orchestration-patterns.md](../references/orchestration-patterns.md) を参照する。

プラグインエージェントは frontmatter の `hooks`、`mcpServers`、`permissionMode` をサポートしない。これらのフィールドは黙って無視される。ここで新しいペルソナを作成するときは、それらへ依存しない。

## 新しいペルソナの追加

1. 既存ペルソナと同じ frontmatter 形式で `agents/<role>.md` を作る。
2. 役割、範囲、出力形式、ルールを定義する。
3. 末尾に **Composition** ブロックを追加する（Invoke directly when / Invoke via / Do not invoke from another persona）。
4. このファイル冒頭の表にペルソナを追加する。
5. そのペルソナが新しいオーケストレーションパターンを可能にする場合は、ペルソナファイル内でパターンを発明せず、`references/orchestration-patterns.md` に文書化する。
