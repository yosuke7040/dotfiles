# スキルの構造

この文書は、agent-skills のスキルファイルの構造と形式を説明する。新しいスキルへ貢献するとき、または既存スキルを理解するときのガイドとして使う。

## ファイルの場所

すべてのスキルは `skills/` 配下の独自ディレクトリに置く:

```
skills/
  skill-name/
    SKILL.md           # 必須: スキル定義
    scripts/           # 任意: スキルワークフローで使う実行可能ヘルパー
    supporting-file.md # 任意: 必要時に読み込む参照資料
```

必須ファイルは `SKILL.md` だけである。`scripts/` は、そのスキルが実際に実行可能ヘルパーを同梱する場合だけ追加し、Markdown だけのスキルではディレクトリごと省略する。

## SKILL.md の形式

### Frontmatter（必須）

```yaml
---
name: skill-name-with-hyphens
description: [task/workflow] を通じて agents を案内する。[specific trigger conditions] の場合に使う。
---
```

**ルール:**
- `name`: 小文字、ハイフン区切り。ディレクトリ名と一致しなければならない。
- `description`: まずスキルが何をするかを三人称で書き、その後に明確な「使う場面」のトリガー条件を 1 つ以上含める。*何を* と *いつ* の両方を含める。最大 1024 文字。

**なぜ重要か:** エージェントは説明を読んでスキルを発見する。説明はシステムプロンプトへ注入されるため、スキルが何を提供し、いつ起動すべきかをエージェントへ伝えなければならない。ワークフローを要約しない。説明に手順が含まれていると、エージェントが完全なスキルを読まずに要約へ従う可能性がある。

### 標準セクション（推奨パターン）

上記の frontmatter 契約は必須である。以下のセクション構成は推奨パターンであり、硬直したテンプレートではない。同じ目的を明確に果たすなら、同等の見出しも許容される。

```markdown
# スキルタイトル

## 概要
このスキルが何をし、なぜ重要かを 1 から 2 文で説明する。

## 使う場面
- トリガー条件（症状、タスク種別）の箇条書き
- 使わない場面（除外条件）

## [中核プロセス / ワークフロー / 手順]
エージェントが従う主ワークフロー。番号付き手順またはフェーズに分ける。
役立つ場合はコード例を含める。
判断点がある場合はフローチャート（ASCII）を使う。

## [具体的な技法 / パターン]
特定シナリオの詳細な指針。
コード例、テンプレート、設定。

## よくある正当化
| 正当化 | 現実 |
|--------|------|
| エージェントが手順を省くために使う言い訳 | その言い訳が間違っている理由 |

## 危険信号
- スキルが破られていることを示す振る舞い
- レビュー時に注意すべきこと

## 検証
スキルのプロセス完了後に確認する:
- [ ] 終了条件のチェックリスト
- [ ] 必要な証拠
```

## セクションの目的

### 概要
スキルの短い説明である。何をするスキルで、なぜエージェントが従うべきかに答える。

### 使う場面
エージェントと人間が、このスキルを現在のタスクに適用すべきか判断する助けになる。肯定的なトリガー（「X のとき使う」）と否定的な除外条件（「Y には使わない」）の両方を含める。

### 中核プロセス
スキルの中心である。エージェントが従う段階的なワークフローを書く。曖昧な助言ではなく、具体的で実行可能でなければならない。

**良い例:** 「`npm test` を実行し、すべてのテストが通ることを確認する」
**悪い例:** 「テストが動くことを確認する」

### よくある正当化
よく作られたスキルの最も特徴的な要素である。重要な手順を省くためにエージェントが使う言い訳と、それに対する反論を組にする。これにより、エージェントがプロセスから逃げるための正当化を防ぐ。

エージェントが「テストは後で追加します」や「これは単純なので仕様は省いてよい」と言った場面を思い出す。そうした言い訳を、事実に基づく反論とともにここへ入れる。

### 危険信号
スキルが破られていることを観察可能に示す兆候である。コードレビューや自己監視で役立つ。

### 検証
終了条件である。エージェントがスキルのプロセス完了を確認するためのチェックリストを置く。各チェックボックスは、テスト出力、ビルド結果、スクリーンショットなどの証拠で検証可能でなければならない。

## 補助ファイル

補助ファイルを作るのは次の場合だけにする:
- 参照資料が 100 行を超える（主 `SKILL.md` を集中させる）
- コードツールまたはスクリプトが必要
- チェックリストが長く、別ファイルにする意味がある

パターンや原則が 50 行未満なら本文内に置く。

スキルが実行可能ヘルパーを必要としないなら、他のスキルに合わせるためだけに空の `scripts/` ディレクトリを作らない。空ディレクトリは、スキルの働きを変えずにノイズだけを増やす。

## 執筆原則

1. **知識よりプロセス。** スキルはワークフローであり、参照ドキュメントではない。事実ではなく手順を書く。
2. **一般論より具体。** 「テストを検証する」より「`npm test` を実行する」のほうがよい。
3. **仮定より証拠。** すべての検証チェックボックスには証拠が必要である。
4. **正当化への対抗。** 省かれがちな各手順には、正当化テーブルで反論が必要である。
5. **段階的開示。** 主 `SKILL.md` が入口である。補助ファイルは必要時だけ読み込む。
6. **トークンを意識する。** すべてのセクションは含める理由を持たなければならない。削ってもエージェントの振る舞いが変わらないなら削る。

## 命名規則

- スキルディレクトリ: `lowercase-hyphen-separated`
- スキルファイル: `SKILL.md`（常に大文字）
- 補助ファイル: `lowercase-hyphen-separated.md`
- 参照資料: スキルディレクトリ内ではなく、プロジェクトルートの `references/` に置く

## スキル間参照

他のスキルは名前で参照する:

```markdown
テストを書くには `test-driven-development` スキルに従う。
ビルドが壊れたら `debugging-and-error-recovery` スキルを使う。
```

スキル間で内容を重複させない。代わりに参照し、リンクする。

## 必須と推奨

必須:

- `skills/<skill-name>/SKILL.md` ファイル
- `name` と `description` を持つ有効な YAML frontmatter
- スキルが何をし、いつ使うかの両方を含む説明

推奨:

- 上に示した標準セクションの流れ
- スキルにとって自然に読める場合は、「仕組み」「中核プロセス」「ワークフロー」のような同等見出し
- 主 `SKILL.md` を集中させる場合だけ補助ファイルを置く
