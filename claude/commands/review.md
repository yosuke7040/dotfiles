code-review-and-quality スキルを起動する。

現在の変更（ステージ済み、または直近のコミット）を、5 つの軸すべてでレビューする:

1. **正しさ**: 仕様と合っているか。エッジケースは処理されているか。テストは十分か
2. **読みやすさ**: 名前は明確か。ロジックは素直か。整理されているか
3. **アーキテクチャ**: 既存パターンに従っているか。境界はきれいか。抽象化の粒度は適切か
4. **セキュリティ**: 入力は検証されているか。シークレットは安全か。認証と認可は確認されているか（security-and-hardening スキルを使う）
5. **性能**: N+1 クエリはないか。上限のない処理はないか（performance-optimization スキルを使う）

指摘は「致命的」「重要」「提案」に分類する。
具体的な file:line 参照と修正提案を含む、構造化されたレビューを出力する。

## Codex との 2 エージェントレビュー

マージ前ゲートとして、Claude の 5 軸レビューに加えて Codex によるレビューも実施し、結果を統合する。これは「両者一致 = 確度高い指摘」「片方のみ = 観点漏れ候補」「対立 = 議論すべきポイント」を浮かび上がらせるためである。

手順:

1. Claude の 5 軸レビュー（code-review-and-quality）を実行して結果を確定させる。
2. その後、`codex-second-opinion` スキルをレビューモードで起動する。request_type は `fix_review` を基本とし、対象は現在の diff（ステージ済み、または直近のコミット）。Codex への質問はレビュー軸を絞らず「Claude が見落としている観点や、別の角度からの指摘を期待」と添える。
3. codex-second-opinion から Codex の指摘が返ってきたら、Claude のレビューと突き合わせて統合する。次の 3 セクションを必ず設ける:
   - **両者一致** — Claude と Codex の両方が指摘した項目。最優先で対応。
   - **片方のみ** — どちらか一方が指摘した項目。観点として有効か個別に判断。
   - **対立・温度差** — 同じ箇所への指摘の重さや方向が異なる項目。ユーザーに議論を提示。
4. codex-second-opinion が利用不可の場合は、その旨を明示して Claude 単独のレビューだけ出力する。

このコマンドはローカル diff に閉じた軽量版である。submodule 跨ぎや PR URL 起点の本格レビューは別途 review-branch skill を使う。
