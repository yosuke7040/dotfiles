shipping-and-launch スキルを起動する。

`/ship` は **ファンアウト型オーケストレーター** である。現在の変更に対して 3 つの専門ペルソナを並列に実行し、その報告をロールバック計画付きの単一の GO/NO-GO 判断へ統合する。ペルソナは独立して動作する。共有状態も順序依存もないため、ここでは並列実行が安全で有用になる。

## フェーズ A: 並列ファンアウト

3 つのサブエージェントを同時に起動する。CLI は `agents/` 内の各カスタムサブエージェントを同名のツールとして公開する。つまり `code-reviewer.md` はメインエージェントが呼び出せる `code-reviewer` ツールになり、プロンプト内の `@code-reviewer` は明示的な起動として機能する。**3 つすべてのサブエージェントツール呼び出しを 1 回の assistant ターンで発行し、並列実行させること**。逐次呼び出しではこのコマンドの目的が失われる。

各ペルソナをツール名で起動する:

1. **`code-reviewer`**: ステージ済み変更または直近のコミットに対して、正しさ、読みやすさ、アーキテクチャ、セキュリティ、性能の 5 軸レビューを実行する。標準レビュー テンプレートで出力する。
2. **`security-auditor`**: 脆弱性と脅威モデルの確認を実行する。OWASP Top 10、シークレットの扱い、認証と認可、依存関係の CVE を確認する。標準監査レポートを出力する。
3. **`test-engineer`**: 変更に対するテストカバレッジを分析する。正常系、エッジケース、エラー経路、並行実行シナリオの不足を特定する。標準カバレッジ分析を出力する。

現在の CLI バージョンでサブエージェントが使えない場合は、各ペルソナのシステムプロンプトをメインコンテキストで順番に起動し、それらの出力を並列で返ってきたものとして扱う。統合フェーズはそれでも機能する。

制約（CLI のサブエージェントモデルに基づく）:
- サブエージェントは分離されたコンテキストループで実行され、このメインセッションにはレポートだけを返す。
- あるペルソナから別のペルソナへ委譲させない。ファンアウトは平坦に保つ。
- 報告を返すだけでなくチームメイト同士が会話する、より豊かなマルチエージェント協働については `references/orchestration-patterns.md` を参照する。

**ペルソナ解決。** `agents/` またはグローバル設定で独自の `code-reviewer`、`security-auditor`、`test-engineer` を定義している場合、それらがこのプラグイン版より優先される。`/ship` はカスタマイズを自動的に拾う。これは意図的な設計である。プラグインのサブエージェントは CLI のスコープ優先順位表の最下位にあるため、ユーザー定義が勝つ。

## フェーズ B: メインコンテキストで統合

3 つのレポートがすべて戻ったら、サブペルソナではなくメインエージェントが統合する:

1. **コード品質**: `code-reviewer` の Critical/Important 指摘と、失敗しているテスト、lint、ビルド出力を集約する。レビュアー間の重複を解決する。
2. **セキュリティ**: `security-auditor` の Critical/High 指摘をリリースブロッカーに昇格する。`code-reviewer` のセキュリティ軸と相互参照する。
3. **性能**: `code-reviewer` の性能軸から取り込む。該当する場合は Core Web Vitals と照合する。
4. **アクセシビリティ**: キーボードナビゲーション、スクリーンリーダー対応、コントラストを確認する。3 つのペルソナでは扱わないため、ここで直接処理するか、アクセシビリティチェックリストを起動する。
5. **インフラ**: 環境変数、マイグレーション、監視、フィーチャーフラグを直接確認する。
6. **ドキュメント**: README、ADR、changelog を直接確認する。

## フェーズ C: 判断とロールバック

単一の出力を作成する:

```markdown
## リリース判断: GO | NO-GO

### ブロッカー（リリース前に必ず修正）
- [情報源ペルソナ: Critical 指摘 + file:line]

### 推奨修正（リリース前に直すべき）
- [情報源ペルソナ: Important 指摘 + file:line]

### 受容済みリスク（それでもリリースする）
- [リスク + 緩和策]

### ロールバック計画
- トリガー条件: [どのシグナルでロールバックするか]
- ロールバック手順: [正確な手順]
- 目標復旧時間: [目標]

### 専門家レポート（全文）
- [code-reviewer レポート]
- [security-auditor レポート]
- [test-engineer レポート]
```

## フェーズ D: Codex による ship 前 sanity check（high stakes 時のみ）

統合判断（フェーズ C）を出す前に、変更の stakes を評価する。次のいずれかに該当する場合は **high stakes** とみなし、Codex への sanity check を追加で挟む:

- 認証、認可、セッション管理に触れる
- 支払い、課金、金額計算に触れる
- データ移行（マイグレーション、バックフィル、削除）を含む
- デプロイ、インフラ、フィーチャーフラグ、シークレットの変更を含む
- ロールバック手順が複雑、または不可逆な操作を含む

low stakes（ロジックの局所修正、UI 微調整、ドキュメント等）ならこのフェーズはスキップして良い。

high stakes と判定したら次を行う:

1. ユーザーに「high stakes な変更のため、Codex の sanity check を挟みます。スキップしますか？」と確認する。明示的にスキップが選ばれた場合のみ省略する。
2. `codex-second-opinion` スキルを起動する。request_type は `fix_review`、対象は現在の diff（ステージ済み、または直近のコミット）。Codex への質問は本番運用観点に集中させ、次のような観点で 5〜7 件構築する:
   - 本番投入後に顕在化しうるリスクは何か
   - ロールバック計画は十分か、また安全に戻せるか
   - 必要な監視・alerting は揃っているか
   - データ整合性、可逆性に懸念はあるか
   - 認証・認可・権限の抜けはないか
3. codex-second-opinion から指摘が返ってきたら、フェーズ C の「ブロッカー」「推奨修正」「受容済みリスク」「ロールバック計画」へ適切に統合する。フェーズ A の 3 ペルソナの指摘と重複する箇所は「両者一致 = 確度高い」として優先する。
4. codex-second-opinion が利用不可の場合は、その旨を明示して 3 ペルソナの結果のみで判断する。

## ルール

1. フェーズ A の 3 つのペルソナは並列に実行する。逐次実行してはならない。
2. ペルソナ同士は互いに呼び出さない。メインエージェントがフェーズ B で統合する。
3. GO 判断の前に、ロールバック計画は必須である。
4. いずれかのペルソナが Critical 指摘を返した場合、ユーザーが明示的にリスクを受け入れない限り、既定の判定は NO-GO とする。
5. **ファンアウトを省略できるのは、次のすべてが真の場合だけである:** 変更が 2 ファイル以下、差分が 50 行未満、かつ認証、支払い、データアクセス、設定や環境変数に触れていない。それ以外ではファンアウトを既定とする。`/ship` は本番投入予定の変更向けに設計されている。影響範囲が自明でない場合、差分が小さく見えても並列レビューを実行する。
