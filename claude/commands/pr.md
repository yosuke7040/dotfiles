# /pr - GitHub Pull Request Description Generator

GitHub の Pull Request description を生成するカスタムコマンドです。

## 使い方

```
/pr [変更の概要]
```

## オプション

- `--simple`: 最小限の項目のみ生成（背景・変更内容のみ）
- `--detailed`: すべての項目を含む詳細版を生成
- `--linear [TICKET-ID]`: Linear チケット ID を自動的に背景に追加
- `--type [feat|fix|refactor|docs|test|chore]`: 変更タイプを指定

## 使用例

```
/pr ユーザー認証機能の実装
/pr --linear ENG-123 ログイン画面のUI改善
/pr --simple バグ修正
/pr --type feat --detailed 新規API実装
```

## 執筆方針

description の目的は「コードを読んでも復元できない情報」を残すこと。長く詳しい description よりも、短くて WHY が伝わる description を優先する。

- **冒頭は見出し配下の箇条書き 3 行**: description の最初は **背景 / 概要に相当する見出し**（例: `## 背景`, `## 概要`, `## Why`, `## Summary` 等）を置き、その直下に `-` で始まる 1 行 1 ポイントの箇条書きを 3 行書く。1 行目に関連 issue / チケットのリンク、2-3 行目に WHY / 何を解決するかを 1 行ずつ簡潔に書く。長文の前置き・段落は避ける。
    - **テンプレートに該当見出しがある場合はそれをそのまま使う**（`## 背景`/`## Why`/`## Summary` など、テンプレートの語彙・見出しレベルを尊重する）。
    - **テンプレートに該当見出しが無い場合**は、テンプレートのトーンに合わせて `## 背景` または `## 概要` を新規に追加する。
    - 見出し名を勝手に置き換えたり、複数の類似見出しを増やしたりしない。
- **HOW は最小限に**: どのファイルで何をどう変えたかの網羅的な列挙は書かない。差分・コミットログで足りる情報を description に複写しない。
- **WHY を厚く**: なぜこの変更が必要か、何を解決するのか、を冒頭で明示する。
- **判断と選択肢**: 実装過程で検討した代替案、却下した理由、トレードオフを残す。`<details><summary>判断と選択肢</summary>...</details>` で**畳んで残す**。冒頭の見通しを保ちつつ、開けば判断の経緯がわかる状態にする。検討が無かった場合は無理に書かない。
- **コードに任せる**: 変更ファイル一覧、行数、内部実装の詳細はリンク・差分に任せる。description に貼り直さない。

## コマンドの実行方法

このコマンドは以下の手順で動作します：

1. デフォルトブランチ（main/master）を自動検出
2. 現在のブランチとデフォルトブランチの差分を確認
3. 変更されたファイルの内容を分析
4. コミットメッセージから変更の意図を理解
5. 適切な PR description を生成
6. draft PR として作成（`gh pr create --draft`）
7. **作成後にセルフレビュー**し、必要に応じて `gh pr edit` で description を修正

### 実行時の処理フロー

1. **デフォルトブランチの検出**

   ```bash
   # リモートのデフォルトブランチを取得
   git symbolic-ref refs/remotes/origin/HEAD | sed 's@^refs/remotes/origin/@@'
   # または git rev-parse --abbrev-ref origin/HEAD
   ```

2. **差分の取得**

   ```bash
   # デフォルトブランチとの差分を確認
   git diff <default-branch>...HEAD
   git log <default-branch>..HEAD
   ```

3. **PR テンプレートの検索と適用**

   ```bash
   # 以下の順序で PULL_REQUEST_TEMPLATE.md を探す
   .github/PULL_REQUEST_TEMPLATE.md
   PULL_REQUEST_TEMPLATE.md
   docs/PULL_REQUEST_TEMPLATE.md
   ```

   - **テンプレートが見つかった場合**: テンプレートの構造・セクションに従って PR description を生成する
   - **テンプレートが見つからない場合**: PR を作成する前に、生成した description の内容をユーザーに提示し、確認・承認を得てから作成すること（自動で作成してはならない）

4. **変更内容の分析**
   - 変更されたファイルのリスト
   - 各ファイルの変更行数
   - コミットメッセージの集約
   - ブランチ名からの文脈推測

5. **PR 作成後のセルフレビューと修正**

   draft PR を作成したら、**そこで終わらず**自分が書いた description を読み返し、必要があれば修正する。GitHub 上で実際にレンダリングされた状態を前提にレビューする。

   レビュー観点（このコマンドの「執筆方針」「生成時の考慮事項」に照らす）:
   - 冒頭が **背景 / 概要に相当する見出し**（テンプレートがあればその見出し、無ければ `## 背景` 等）の直下に置かれた `-` 箇条書き 3 行に収まり、1 行目に Linear / Issue リンクが貼られているか
   - テンプレートの見出し名・見出しレベルを勝手に変更していないか
   - **判断と選択肢** が `<details><summary>判断と選択肢</summary>...</details>` で畳まれているか
   - **HOW の冗長な列挙**（ファイル一覧・行数・実装手順の逐次説明など）が混入していないか
   - **空セクション**や「該当なし」だけのセクションが残っていないか
   - 機密情報・個人情報・内部 URL が混入していないか
   - 不要な絵文字や定型句（生成 AI のクセが残ったもの）が無いか
   - Markdown の崩れ、リンク切れ、誤字脱字が無いか

   修正が必要な場合の手順:

   ```bash
   # 現在の description を取得して内容を確認
   gh pr view <PR番号> --json body --jq .body

   # description を編集（エディタが開く、または --body / --body-file で直接渡す）
   gh pr edit <PR番号> --body-file <修正後の本文ファイル>
   # 部分修正なら一度ファイルに保存 → 修正 → --body-file で上書き
   ```

   修正不要と判断した場合でも「セルフレビューを実施し、修正不要と判断した」ことをユーザーに一言報告する（黙って終わらない）。

### 生成時の考慮事項

WHY を厚く、HOW を薄く。下記の項目はテンプレートではなく観点であり、**該当が無ければ無理に埋めず省略する**。空セクションを機械的に作らない。

- **背景 / WHY**: description 冒頭に置く。**背景・概要に相当する見出し**（テンプレートに該当見出しがあればそれを使い、無ければ `## 背景` を追加）の直下に、**`-` で始まる箇条書き 3 行**（1 行 1 ポイント）で、1 行目に関連 issue / チケットのリンク、2-3 行目に「なぜこの変更が必要か」「何を解決するか」を簡潔に書く。最優先で書く項目。
- **判断と選択肢**: 検討した代替案、採用しなかった理由、トレードオフ。`<details><summary>判断と選択肢</summary>...</details>` で**畳んで残す**。冒頭の見通しを優先し、開けば読める状態にする。
- **変更内容（要点のみ）**: 概観として 2-4 行程度。ファイル一覧や逐次的な変更手順は書かず、差分を読む前の見取り図に留める。
- **スコープ外**: 意図的に含めなかった変更や、別 PR で対応予定の項目。
- **後続タスク**: この PR のマージ後に必要となる作業。
- **参考情報**: 関連するドキュメント、議論のリンクなど。

### 自動生成される内容

以下の情報は自動的に取得・生成されます：

- デフォルトブランチ（main/master）の自動検出
- 変更されたファイルのリスト（`git diff --name-status`）
- 追加・削除された行数（`git diff --stat`）
- コミットメッセージの要約（`git log --oneline`）
- ブランチ名からの推測（feature/login → ログイン機能関連など）
- ファイルタイプからの推測（\*.test.js → テスト追加など）

> これらは PR の意図を推測するための**内部情報**として使用する。description 本文にはそのまま貼り付けない（差分・コミット履歴で見れば足りるため）。WHY を組み立てるための材料として活用する。

### デフォルトブランチとの差分確認

このコマンドは以下の方法でデフォルトブランチを検出し、差分を確認します：

```bash
# デフォルトブランチの自動検出（優先順位）
1. git symbolic-ref refs/remotes/origin/HEAD
2. git config init.defaultBranch
3. main または master の存在確認

# 差分の確認コマンド例
git diff origin/main...HEAD --name-status
git log origin/main..HEAD --oneline
```

### 注意事項

- **⚠️ PR は必ず draft（下書き）として作成すること。`gh pr create --draft` を使用し、通常の PR として作成してはならない。レビュー準備が整ったらユーザー自身が draft を解除する。**
- **⚠️ draft PR 作成後は必ずセルフレビューを行い、必要に応じて `gh pr edit` で description を修正すること。作成して終わりにしない**（観点と手順は「実行時の処理フロー」の 5 を参照）。
- 作業ブランチから実行してください（デフォルトブランチでの実行は避けてください）
- リモートの最新情報を取得するため、実行前に `git fetch` を推奨
- プライベートな情報が含まれていないか確認してから PR を作成してください
- コードブロック（` ```markdown ` ～ ` ``` `）で囲んで出力してください（VSCode でコピーしやすくするため）
- description は短い方がよい。HOW を増やしてページを長くするより、WHY を 2-3 行追加する方が価値が高い
