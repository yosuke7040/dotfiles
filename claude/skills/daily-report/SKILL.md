# デイリーレポート生成

Claude Code の会話履歴、GitHub アクティビティ、Slack メッセージからアクティビティレポートを生成する。

**重要な出力ルール**: 生成されるレポートファイルには「Claude」「Claude Code」「sessions」「history.jsonl」やデータ収集方法に関する実装の詳細を絶対に記載しないこと。レポートはソースに依存しない、クリーンなアクティビティログとして読めるようにする。このルールは出力されるレポートにのみ適用され、このスキル手順書では Claude の内部仕様を自由に参照してよい。

## 使用タイミング

- 日次/週次サマリー
- スタンドアップの準備
- ステークホルダーへの状況報告
- プロジェクト横断の成果追跡

## オプション

- `--days N` または数値: 過去N日分（デフォルト: 当日）
- `--yesterday`: 昨日のみ
- `--project NAME`: プロジェクトでフィルタ
- `--summary`: プロジェクトサマリーテーブルを追加

## 日付範囲リファレンス

オプションから `{date}` と `{next_date}` を決定する。今日の日付は `date +%Y-%m-%d` で取得。

| オプション       | `{date}`   | `{next_date}` |
| --------------- | ---------- | ------------- |
| デフォルト（当日）| 今日       | 明日          |
| `--yesterday`   | 昨日       | 今日          |
| `--days N`      | N日前      | 明日          |

## 手順

### 並列初期化

データ収集前に、ツールの利用可否を並行してチェックする（可能な限り並列化）:

```bash
# 3つとも同時に実行
gh api /user --jq '.login'      # gh が認証済みか確認; gh_user として取得
gws calendar +agenda --today    # gws が利用可能か確認（利用不可の場合はメモ）
slack_read_user_profile          # Slack MCP が接続されているか確認（利用不可の場合はメモ）
```

結果に基づき、利用不可なデータソースは後続のステップでスキップする。

### Claude アクティビティの集約

**場所:** `~/.claude/history.jsonl`

このファイルが存在しないか空の場合、このステップをスキップして GitHub アクティビティに進む。

**フォーマット:** JSONL、以下のフィールドを含む:

- `timestamp`: Unixミリ秒 → ローカル日付に変換
- `project`: フルパス → 最後のセグメントをプロジェクト名として抽出
- `display`: ユーザーのメッセージ
- `sessionId`: 関連する作業をグループ化

#### エントリの読み取り

1. 行数を取得: `wc -l ~/.claude/history.jsonl`
2. 大きめのバッファを読み取る: `offset=(lines - 3000)` で約2週間分のアクティビティをカバー
3. タイムスタンプでエントリをフィルタ — `timestamp`（Unixミリ秒）をローカル日付に変換し、対象日付範囲と比較、範囲外のエントリを除外

#### プロジェクトパスからGitHubリポジトリを抽出

`ghq` 規約を使用するプロジェクトパスには `owner/repo` が含まれる:

```
/Users/*/ghq/github.com/{owner}/{repo} → {owner}/{repo}
```

パターンで抽出: `github.com/` にマッチし、次の2つのパスセグメントを取得。

- `owner/repo` → プロジェクト表示名のマップを構築
- GitHub以外のプロジェクトパス（例: `~/notes/project-x`）にはリポジトリがない — レポートにはローカルプロジェクトとして残し、GitHubクエリではスキップ

#### ノイズの除去

以下を除去:

- スラッシュコマンド（`/clear`、`/model` 等）
- 単一応答: "yes"、"no"、"1"、"2"、"a"、"b"
- 空のエントリ/重複エントリ

#### セッションごとの要約

同じ sessionId のエントリを意味のある要約にグループ化:

| 生のメッセージ                       | 要約                           |
| ---------------------------------- | ----------------------------- |
| "fix lint", "yes", "run tests"     | lint修正とテスト作業            |
| "implement X", "yes", "continue"   | X機能の実装                    |
| "investigate bug", "this error..." | バグの調査                     |

#### カテゴリ分類

以下のタグを付与: **feat**, **fix**, **refactor**, **docs**, **test**, **ci**, **plan**

### 出力フォーマット

```yaml
---
date: YYYY-MM-DD
type: daily-report
---
```

```markdown
# デイリーアクティビティレポート YYYY-MM-DD

## カレンダー

- 14:30 デザインレビュー
- 16:00 マネージャーとの1on1

## owner/repo-name

ユーザー認証を実装し、ページネーションのバグを修正。ログアウトボタンの追加と
空状態の表示を改善。

- PR #12 ステータスラインのスタイル追加 [merged]
- Review #45 実装のクリーンアップ [APPROVED]

## notebook

ミーティングノートとスニペット変換の作業。

## Slack

- **#eng**: デプロイロールバック戦略の議論とポストモーテムの共有
- **#project-x**: デザインモックアップのレビューとv2ローンチのタイムライン確認

## その他のリポジトリ

- Review myorg/repo#5 time依存関係の更新 [APPROVED]
- Issue myorg/other-repo#10 バグ報告 [open]

## フィードバック

- **Slack**: #eng での障害報告に対して「修正しました」とだけ返信していたが、何が原因で何を修正したかを1〜2文で添えると、チームの学びになる。例: 「キャッシュTTLの設定ミスが原因でした。TTLを300s→60sに変更して解消」
- **開発**: PR #12 の説明が空だった。変更の背景と影響範囲を書くと、レビュアーの負担が減りレビュー速度が上がる
```

**プロジェクトごとのフォーマット:**

1. **要約**（必須）: すべてのソース（Claude履歴、コミットメッセージ、PR説明）から統合した1〜3文。散文として記述し、タグやプレフィックスは付けない。データの収集方法ではなく、_何をしたか_を記述する。
2. **PRとレビュー**（任意）: 存在する場合のみ要約の下に記載。サブヘッダーなし — 項目のみ。フォーマット: `- PR #N タイトル [state]` または `- Review owner/repo#N タイトル [STATE]`。
3. PRやレビューがないプロジェクトは要約のみ。

**その他のルール:**

- Claude履歴からのGitHub以外のプロジェクトは、ディレクトリ名をセクションヘッダーとして使用（例: "notebook"）。
- 既知のプロジェクトに一致しないGitHubの項目は「その他のリポジトリ」に配置。
- アクティビティがないセクションは省略。
- `[tag]` プレフィックス、コミットハッシュ、「N commits pushed」は不要。

### フィードバック生成

レポートの全データ（Slack メッセージ、PR、コミット、レビュー、カレンダー）を分析し、シニアエンジニアの視点から改善提案を生成する。

**分析の観点:**

| 観点 | チェックポイント |
| --- | --- |
| Slack コミュニケーション | 返信の情報量は十分か、スレッドを適切に使っているか、適切なチャンネルで発言しているか、技術的な説明が簡潔で正確か |
| 開発業務 | PR の粒度は適切か、PR説明・コミットメッセージは明確か、レビューコメントは具体的か、タスクの優先順位は妥当か |
| 時間の使い方 | コンテキストスイッチが多すぎないか、集中作業の時間を確保できているか、ミーティングとのバランスは取れているか |

**ルール:**

- 指摘がない場合は「フィードバック」セクション自体を省略する（無理に指摘を作らない）
- 各指摘には具体的なアクションを含める（「〜すべき」ではなく「〜するとよい。例えば〜」の形式）
- 批判ではなく建設的な提案として書く
- 実際のデータに基づく指摘のみ行う（推測で指摘しない）

### 出力パス

1. **レポートファイルを書き込み**: カレントディレクトリに `report-YYYY-MM-DD.md`
   - 複数日レポートの場合: `report-YYYY-MM-DD-N-days.md`
2. **要約**: 書き込み後、ファイルパスをユーザーに伝え、何が記録されたかの概要を報告する。

## サマリーテーブル（--summary 使用時）

```markdown
| プロジェクト | アクティブ日数 | アクティビティ | PR数 | レビュー数 | 主な作業 |
| ----------- | ------------ | ------------ | ---- | --------- | ------- |
| my-proj     | 5            | 45           | 3    | 2         | feat, fix |
```
