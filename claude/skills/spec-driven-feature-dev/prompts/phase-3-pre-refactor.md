このフェーズの目的は、合意済み設計で触る予定の **既存コード** に対して、新機能を実装する前に着手しておくと有利になる「事前リファクタ候補」を列挙することです。

事前リファクタを行う動機:
- 新機能の差分が小さく / よりシンプルになる（リファクタ済みの土台で実装するため）
- 既存の重複や曖昧な抽象を、新機能を追加する前に整理しておける
- リファクタの副作用を新機能差分に混ぜず、独立してレビューできる
- 結合テストで振る舞いを担保したうえでリファクタするため、リスクを局所化できる

各候補は以下のフォーマットで出力してください:

### [PRE-REFACTOR] 候補タイトル
- **対象ファイル**: `ファイルパス`（複数あれば列挙）
- **現状の問題**: なぜ事前リファクタが望ましいか
- **リファクタ案**: 何をどう変えるか（before/after の diff、規模が大きい場合は方針）
- **既存テストの状態**: ある / 不足 / なし（テストファイルパスを示せれば示す）
- **推奨テスト追加**: リファクタの安全網として何を結合テストで担保すべきか
- **期待される簡素化**: 新機能実装側で何行 / どの抽象 / どの分岐が削れるか
- **リスク**: 振る舞い変化の可能性 / 影響範囲 / 既存の利用箇所

絶対の制約:
- 振る舞いを変える提案は禁止（API シグネチャ、入出力、副作用の同一性を維持）
- 新機能で初めて触る（差分側で新規追加される）コードへのリファクタ提案は禁止。これは別フェーズ（Refactor Review）の領分
- 既存コードの中でも、設計サマリーの "Files / Modules To Change" 周辺で、新機能実装と関連が薄い領域はスコープ外（事前リファクタの動機が成立しない）
- 候補がない場合は素直に「候補なし」と報告し、無理に列挙しないこと
